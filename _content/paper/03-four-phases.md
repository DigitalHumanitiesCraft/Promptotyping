# 3. The Four Phases of Promptotyping

## 3.1 Preparation

The Preparation phase collects and structures all materials before technical decisions are made. This includes research data in their original formats, documentation of standards and data models, research questions and problem statements, and domain knowledge — whether from publications, expert interviews, or prior project experience. The phase forces precise articulation of project goals before LLM use; vagueness here cascades through all subsequent phases.

The phase typically begins with requirements engineering — articulating user stories, data-to-question mappings, and analytical perspectives, whether from the researcher's own domain knowledge or from collaborative Scholar-Centred Design sessions (Section 2.5). In the DEPCHA project, deep-dive sessions with historians of different source types (tax registers, monastic accounts, merchant day books) surfaced recurring representation needs that could not have been derived from the data alone (Pollin 2025b). In simpler projects, the researcher serves as both domain expert and requirements analyst.

Across all documented projects, a consistent repository architecture has emerged. This is not prescribed but has proven stable across diverse project types:

```
project-root/
├── knowledge/          # Promptotyping Documents
├── data/               # Source data and intermediate outputs
├── scripts/            # Processing scripts (often LLM-generated)
├── docs/               # Web frontend / publication outputs
├── pipeline/           # Transformation and validation logic
└── CLAUDE.md / .cursorrules  # Agent configuration
```

The `CLAUDE.md` file at the repository root deserves particular attention. It configures the agent's relationship to the project by defining workflow conventions, security constraints, code style preferences, and project-specific rules. In mature projects, this file functions as what we call _agent socialisation_ — not merely instructing the agent what to do but embedding it with a value system that guides decisions in unforeseen situations. A `DESIGN.md` file serves a similar dual function: it documents design rationale for human readers while simultaneously configuring the agent's aesthetic and interaction decisions.

The Preparation phase ends when the repository structure exists, source data is accessible, initial research questions are formulated, and requirements are documented — not as a comprehensive specification but as a structured starting point for exploration.

## 3.2 Exploration and Mapping

The Exploration phase is a systematic investigation of the interface between data and research context. The central question is: can the abstract research question be concretely mapped to the available data structure? The LLM is used to generate mapping hypotheses — which data fields correspond to which analytical categories, which visualisation types fit the data characteristics, and where the data is insufficient for the intended purpose.

This phase has an epistemic dimension beyond efficiency. The LLM generates options that the researcher would not have designed independently. In the CorrespExplorer (Section 4), six visualisation types were explored — map, timeline, Sankey diagram, force-directed graph, matrix heatmap, and chord diagram — before two were discarded as unsuitable for the data and the research question. The researcher's role was not to generate these options but to evaluate them against domain criteria. This is a different cognitive mode than classical software development, where one must conceive a solution before implementing it. In the DHd Landscape project (Section 4), the initial idea was deliberately vague ("look at DH labs, build an information visualisation"); the structure — scope, data sources, technology, interface paradigm — emerged through conversational exploration with the LLM.

Understanding what the data _cannot_ support is as valuable as knowing what is possible. Dead ends are positive findings, not failures — they prevent reinvestigating failed approaches in later phases.

The Exploration phase ends when there is a clear understanding of what is feasible, what is not, and why — documented sufficiently that the reasoning is recoverable.

## 3.3 Distillation

The Distillation phase compresses the knowledge gained during Exploration into structured Markdown documents optimised for LLM processing. The core principle, motivated by the empirical findings on Context Rot (Hong et al. 2025), is _maximum information with minimum tokens_. Hong et al. demonstrate that LLM performance degrades non-uniformly as input length increases, even on simple retrieval tasks. This finding provides the technical rationale for aggressive compression: documents must be structured so that the most critical information occupies the positions where model attention is highest (typically the beginning and end of the context), while redundancies are eliminated.

The process of compression is not neutral. Context Rot describes, in information-theoretic terms, noise in the communication channel between document and model. Distillation is an encoding strategy that optimises the signal-to-noise ratio within this channel. What makes this consequential is that the encoding decisions are epistemic decisions: they determine what information will be available to the model in subsequent processing steps and what will not. A Knowledge Document is therefore not a summary of what was learned during Exploration but a translation — a selective transformation that shapes the possibility space of all downstream outputs.

This principle is most visible in the extraction of context knowledge through media change. Context information is initially bound in a medium — a digitised manuscript, a screenshot, a handwritten source. Through the LLM as extraction instrument (typically operating as a vision model), it is transferred into a different medium: a structured Markdown document, a specification, a data model. This extraction is neither lossless nor neutral. What a vision model recognises in a digitised page depends on the prompt. The same screenshot yields a layout document, a colour table, or a component structure depending on the question asked. The extraction produces the knowledge it appears merely to transcribe.

The `knowledge/` directory of a Promptotyping repository functions as a simplified research vault — structurally analogous to a personal Obsidian-style knowledge graph but project-centred and reduced in scope. What constitutes the vault is not a fixed catalogue of files but four operative principles: Markdown with frontmatter, wikilinks as connective tissue, function before filename, and curatable as a network rather than a table of contents. The number of documents varies with project complexity — three in a small tool repository, thirteen in a multi-stage pipeline like ZBZ OCR/TEI, around thirty in VetMedAI Wissensbilanz. Promptotyping does not invent a new document genre; it ports the research vault into the repository, where the same Markdown-and-wikilink practice that researchers cultivate for their personal knowledge becomes the working memory of an agentic project. This portability is also a pedagogical asset: practitioners who already maintain a research vault have internalised the relevant reading heuristic — function over filename, link over hierarchy — and recognise the `knowledge/` folder as a familiar object in a new operative role.

Within this vault, three structural commitments shape what counts as a well-formed Promptotyping knowledge base, independently of how many documents a particular project carries. The first is _function before filename_: a document is identified by the function it serves, not by the filename it bears. The same function — say, describing the project's data substrate — may be carried by `data.md` in one repository, `corpus.md` in an edition project, or `material.md` in another; conversely, the same filename may carry different functions across projects, as `pipeline.md` does in projects where it documents technical realisation versus those where it documents the build sequence. The Critical Expert reads the function, not the listing. The second commitment is _inclusion by trigger, not by checklist_: each function is included in a project only when a trigger criterion holds for that project. Identity is always present; description of material is present when the project processes data; description of form is present when the project has a user interface; navigation infrastructure is present when the vault grows beyond a small set of documents. A small tool repository may consolidate identity, material, and substance into a single `project.md`; a multi-stage pipeline may split substance across `requirements.md`, `features.md`, and `decisions.md`, and split construction across `architecture.md` and `pipeline.md`. What matters is that every function relevant to the project is addressed at exactly one findable location — not that a prescribed set of files is present. The third commitment is _diagnostic decoupling_: the diagnostic question "where do I look when output is wrong?" is answered by the document type (Knowledge for incorrect content, Action for incorrect form, Process for unclear decision logic), not by the function. Function says where information lives; type says how to read it.

These three commitments describe the practice that has emerged across the documented projects. What they leave open is the list of documents a vault carries, not its whole normative content: the one prescribed minimum is a small frontmatter core — `title`, `project`, `method`, `status`, `created`, `updated` — that every hand-authored document shares, and against which a document can be audited. The set of functions that a Promptotyping vault may carry, by contrast, is open: as new project types enter the practice, new functions accrue (an editorial-guidelines function in scholarly editions, an organisational function in multi-agent projects — Section 3.5 — that did not exist in earlier projects). What persists is the form of the argument, not the count of its instances.

Whether these recurring functions could harden into a shared metadata standard was left open in the earlier method description (Pollin 2025d). A partial answer has since taken concrete form. The functions have been consolidated into a versioned, publicly published catalogue of document templates, hosted on the method's documentation site (https://dhcraft.org/Promptotyping/). The catalogue is itself an application of the method to its own artifacts: rather than positing what a Promptotyping document must contain, each template was extracted from the recurring form a function takes across at least two repositories, and its requirements were checked against the empirical record instead of prescribed in advance.

The clearest instance of this self-correction is the frontmatter core described above. An earlier specification required eight mandatory fields; a survey of the real knowledge bases found that not a single repository satisfied that core in full, and that one of its fields existed as frontmatter in exactly one project. The mandatory core was therefore reduced to the six fields the practice actually carries, on the principle that fewer fields are maintained more consistently than more. The reduction corrects the specification rather than relaxing it: a requirement that the practice silently ignores documents nothing.

Each template carries a permanent public address, and a repository links to the authoritative specification through a `template:` field in its document frontmatter. Because the site's browser-facing routes resolve only through client-side JavaScript, the canonical address for machine retrieval is a static Markdown URL, following the pattern `https://dhcraft.org/Promptotyping/_content/promptotyping-document/{slug}.md`, which returns the template specification as raw text. A coding agent that encounters such a field can retrieve the full specification without prior knowledge of the project — the repository explains itself to the agent that reads it.

The status vocabulary was corrected the same way. The maturity values inherited from personal knowledge management (`draft`, `complete`, `reviewed`) did not describe two document kinds that the practice articulates pervasively: the continuously updated process document that is never `complete` by design, such as a journal or a plan, and the dated snapshot whose content is meant to age from the moment of writing, such as a report or a status finding. Two values, `active` and `snapshot`, were added to name them, and the free-form values that repositories had improvised were normalised onto the extended vocabulary.

The catalogue also promoted a practice into a document function. Several projects had begun to check their own empirical and novelty claims against the raw data before those claims were used externally, in a paper, a report, or a handover. This adversarial self-verification, first observed as an ad hoc habit, is now a distinct document function with its own template. We call the resulting documents _verification documents_: they record what claim is checked, against what evidence, by what procedure, and with what verdict. They belong to the Distillation vault and should not be conflated with the Verification Milestones and Verification Interfaces of Section 5, which are checkpoints and browser-based tools in the Implementation pipeline. A verification document is the written audit of a project's own assertions; a Verification Milestone is a stage in the data pipeline where that and other checks are applied.

Three analytically distinct types of Promptotyping Documents have emerged across all documented projects.

**Knowledge Documents** are declarative. They describe what is known about the data, the domain, and the research context. Examples include `DATA.md` (data structure, field definitions, relationships between datasets, example records), `specification.md` (user stories with acceptance criteria, functional and non-functional requirements, and design decisions), `editorial-guidelines.md` (annotation conventions for digital editions), and `mapping.md` (correspondence rules between source formats and target structures). These documents are curated by the Critical Expert and constitute the primary persistent artifact of the context engineering process — the knowledge they encode survives changes in code, model versions, and implementation tools.

**Process Documents** are chronological or analytical. They describe the working process and its outcomes. The central document is `JOURNAL.md`, a session-by-session log of decisions, observations, turning points, and dead ends. In the imareal-room-object project, the LLM agent began documenting autonomously in the journal without developer correction — an early instance of what we call _Context Memory_, where the interplay of `journal.md` (documenting why) and git history (documenting what) enables reliable session resumption even after extended breaks. The VetMedAI project introduced `LEARNINGS.md` as a distinct subcategory: transferable insights distilled from the journal, curated as the project progresses rather than written retrospectively.

**Action Documents** are imperative. They describe what agents can do within the project. The standard Action Document, present in every project, is the `CLAUDE.md` agent configuration in the repository root. Multi-agent setups add further Action Documents specific to orchestrating several parallel agents with differentiated roles: `INSTRUCTIONS.md` (implementation steps), `RULES.md` (global development principles and constraints), and `cloud-commands.md` (available scripts and their invocation patterns for sub-agents). These are not a standard for ordinary single-agent repositories. The category was not present in the initial method description (Pollin 2025d) and reflects the increasing role of sub-agent orchestration in complex projects.

The distinction is analytical, not rigid, but the typological boundary between Knowledge and Action carries weight. A `DESIGN.md` is declarative Knowledge — it describes design stance, design system, and interaction patterns for human readers. The imperative translation that steers the coding agent at UI generation lives in the Action layer in the repo root (typically `CLAUDE.md`), which references `DESIGN.md` as the source of values and converts the design stance into imperative design principles. The aesthetic socialisation of the agent is thus a composition effect across two documents, not a hybrid status of one. What matters is that the documents are maintained, that they compress information effectively, and that they are periodically _refactored_ — consolidated through prompts to remove redundancies and integrate insights from later phases.

A second distinction cuts across the three types: documents are either _hand-curated_ or _deterministically generated_. Hand-curated documents — most Knowledge, all Process, and all Action Documents — are written and maintained by the Critical Expert, possibly with LLM assistance, and represent decisions and judgements that cannot be derived from data alone. Deterministically generated documents are rendered from source data by scripts and committed alongside the curated layer. They are Knowledge Documents in their function — they describe the data and the schema — but they are produced by a pipeline rather than by writing. In the ride-static project, eleven Python scripts traverse 107 TEI-XML files plus the project's ODD schema, write structured JSON inventories, and render two Markdown documents from those inventories: `data.md` (corpus structure, anomalies, reference resolution) and `schema.md` (schema customisations, schema-versus-corpus mismatches). The same pattern recurs in projects with large or heterogeneous source data: VetMedAI processes ~80 Excel files into a hierarchical knowledge base before manual curation; FemPrompt SozArb generates corpus manifests from 326 academic papers; klawiter-rescue inventories 6,296 MediaWiki entries. The motivation is direct: full source corpora overflow the context window — passing 107 TEI files plus a schema to the model on every implementation step would defeat Distillation — while a deterministic aggregation, run once and refreshed when the source changes, produces compact Knowledge Documents that fit the context budget and trace cleanly back to the data. Generated documents carry frontmatter that names them as such (`generated:`, `source:`, `inputs:`), are not edited by hand, and are committed so a fresh clone has the structural knowledge base without re-running the pipeline. This positions Distillation not only as a writing act but as a build-time stage: parts of the vault are themselves outputs of a transformation pipeline, governed by the same reproducibility logic as the code they support.

The three-type classification also provides a diagnostic framework. When the LLM produces factually incorrect output, the Knowledge Documents need revision. When the output is formally wrong (incorrect structure, style, or format), the Action Documents need adjustment. When the decision logic is unclear or inconsistent, the Process Documents need to be consulted or updated. This triage accelerates debugging by directing attention to the right document type.

Across 12 inventoried repositories, approximately 140 documents were classified into the three types Knowledge, Process, and Action. The distribution showed the primacy of knowledge formalisation in the method — the more precisely the domain is documented, the more effective the LLM's output. (An earlier inventory recorded a residual 8% as "hybrids" centred on `DESIGN.md`; with the typological refinement above, those documents are reclassified as Knowledge with an Action-layer companion in the repo root.)

The Distillation phase ends when the documents compress project knowledge sufficiently for implementation — when a new agent instance, given only the documents and the data, could reconstruct the project's logic without additional explanation.

## 3.4 Implementation

The Implementation phase is iterative development with the LLM, using the Promptotyping Documents and the data as context. The approach differs fundamentally from classical software development: the researcher does not write code but steers its generation through structured context and evaluative feedback. Section 4 illustrates these practices across projects of increasing complexity.

The primary implementation instrument is a multi-agent coding tool operating within the project repository. In the documented projects, this is typically Claude Code (Anthropic 2025), though the method is not bound to a specific tool. The agent has access to the full repository through the file system: code, research data, knowledge documents, and scripts. In mature projects, the agent functions not only as a code generator but as the primary interface to the repository: the researcher asks questions through natural language, and the agent answers by reading knowledge documents, navigating files, and executing commands. In the Medieval Legal Transactions project (Section 4), the editor's workflow is entirely mediated through the agent — placing new TEI-XML documents, validating schemas, running regression tests, building the frontend, and verifying results in the browser.

A precondition for effective Promptotyping is that the research data carries semantic structure. All documented projects work with semantically enriched formats such as TEI-XML, RDF, JSON-LD, CMIF, LIDO, and PAGE XML. The LLM can work productively with this data because standards and ontologies already encode domain meaning — they provide the shared vocabulary that bridges between the researcher's intent (expressed in Promptotyping Documents) and the generated code. Unstructured data can be used but requires significantly more effort in the Distillation phase to make the structure explicit.

Implementation follows an iterative cycle with three feedback mechanisms. _Deterministic feedback_ comes from schema validation (TEI-XML against RelaxNG, Schematron), test suites, linting, and build processes — automated checks that produce unambiguous pass/fail results. _Visual feedback_ comes from screenshots of Promptotyping Interfaces, which the LLM can interpret through multimodal capabilities to identify layout errors, rendering problems, or incorrect data mappings. _Expert feedback_ comes from the Critical Expert's evaluation of whether the output is factually correct, domain-appropriate, and aligned with the research question — the only mechanism that can detect confabulation or assess whether the right questions are being asked.

At defined points in the pipeline, progress is paused for systematic verification at what we term _Verification Milestones_. These milestones combine all three feedback mechanisms and serve both human experts and AI agents. Their function is to interrupt the error cascade that characterises multi-stage automated pipelines, where mistakes at one stage propagate and become harder to identify downstream. We call the browser-based tools built for this purpose _Promptotyping Interfaces_: static HTML/CSS/JavaScript applications that render intermediate or final results for human inspection. Five categories have emerged, described in Section 5.3.

Complex projects benefit from running multiple agent instances in parallel. In the Kulturpool Explorer, two Claude Code instances worked simultaneously — one on data harvesting (API exploration, data transformation), the other on design research (visual references, interaction patterns, information architecture) — with knowledge merged into shared documents before the implementation began. This is not delegation in the classical sense but _parallel context exploration_, where each agent develops a different facet of the project simultaneously. The cognitive demands of monitoring parallel agents are significant and explicitly noted in the process documentation as unsustainable for extended periods (Section 6.4).

The `journal.md` documents each implementation cycle chronologically. Git commits function as implicit Process Documents, marking verified states and enabling rollbacks. New knowledge produced during implementation — unexpected data characteristics, performance constraints, design refinements — is written back into the Promptotyping Documents, which are living artifacts rather than static specifications. The output of an Implementation cycle is a functional prototype. Multiple iterations are typical; the Lucina digital edition went through five iterations with separate documentation per iteration and a multi-LLM workflow (Claude for structure, Gemini for long contexts, GPT for corrections).

The Implementation phase does not have a single exit criterion but a spectrum: from a rapid functional prototype (the Stefan Zweig annotation tool in two hours) to a production-grade system (the ZBZ pipeline with 30+ scripts and Verification Interfaces at nine pipeline stages). What determines the appropriate stopping point is the research question, not the technical possibilities.

## 3.5 Two Modes: Promptotyping a Tool and Promptotyping a Research Project

Promptotyping operates in two modes, distinguished not by project size but by the organisational structure of the work.

The **single-agent mode** — _Promptotyping a Tool_ — is the default form documented in Sections 3.1 to 3.4. A single researcher, working with a single Claude Code instance, builds a research artifact: an interface, a pipeline, an edition, a data model, an annotation schema. The Critical Expert in the Loop is the researcher; the Action Layer is `CLAUDE.md`; the vault is read by one agent and curated by one human. Most projects in Section 4 operate in this mode, including the documented progression from the Stefan Zweig annotation tool (~2 hours) to the coOCR-HTR workbench (~17,000 lines, 567 tests).

The **multi-agent mode** — _Promptotyping a Research Project_ — emerges when the work itself has the structure of a research organisation: multiple parallel tasks, differentiated roles, formalised handovers, shared and separate knowledge zones. Here the Action Layer is no longer a single configuration file but an _organisational prompt_. A **Research Lead Agent** — itself a Claude Code instance with a dedicated system prompt — coordinates a small team of Sub-Agents to whom delimited tasks are delegated, whose outputs are reviewed, and whose work is reintegrated into the shared vault. The Kulturpool Explorer (Section 4.2, Insight 8) demonstrated an early form, with two Claude Code instances working in parallel on data harvesting and design research, knowledge merged into shared documents before implementation began. The wiiw-figaro subagents project structures this further with role-specific configurations under `.claude/agents/` — one read-only for analysis, one with full write access for implementation, one for synthesis — each with a defined permission scope and handover protocol. The methodological consequence — that the Action Layer becomes organisation-modelling rather than agent-configuration — has not been previously articulated.

The doctoral analogy is operative, not metaphorical. The Research Lead Agent assigns delimited tasks, defines what the Sub-Agent may decide and what must be returned, reviews outputs before they accumulate into the shared knowledge base, and reassigns when an approach proves unworkable. Sub-Agents have limited write permissions, defined knowledge zones, and report back through structured handovers. This is academic research organisation — the way a principal investigator coordinates doctoral students, masters' candidates, and student assistants — ported into agentic coding. The Critical Expert in the Loop remains the human researcher, but the role widens: in addition to evaluating outputs, the Critical Expert now designs the organisation itself — which Sub-Agents exist, what each may decide, where their knowledge zones overlap and where they must remain separate, and how their outputs are reintegrated into the shared vault. The methodological burden shifts from steering one agent to designing a small research team.

The Action Layer of a multi-agent project is therefore stratified. The repo-root `CLAUDE.md` configures the Research Lead Agent, naming the Sub-Agents available to it and the protocols by which they are invoked. Sub-Agent configurations under `.claude/agents/` (or an equivalent location) define each role, its permissions, and the knowledge zones it may read and write. An organisation document — `agents.md`, `team.md`, or `organisation.md` in the repo root — describes the team as a whole: the rationale for the role split, the handover rules, the cases in which the Research Lead Agent escalates to the human Critical Expert. The Knowledge layer in `knowledge/` remains shared by default, but specific documents may be marked as Sub-Agent-private when the role's information scope is intentionally narrow (for example, an analysis Sub-Agent that should not see prior implementation choices, to keep its analysis from being framed by them).

The two modes are not exclusive. A research project may begin in single-agent mode and shift into multi-agent mode when complexity demands it; a multi-agent project may have phases — initial Exploration, final Distillation — where one agent is sufficient and the team contracts. What the distinction makes visible is that the threshold between the two modes is itself a methodological decision, driven by task structure rather than by tool capability. The four phases (Sections 3.1 to 3.4) apply to both modes; what differs is who executes them. Preparation in multi-agent mode includes designing the Research Lead Agent's organisational prompt; Distillation includes drafting Sub-Agent role definitions; Implementation runs as a coordinated sequence rather than a single iterative loop. Section 6.2 develops the autonomy-zone implications of this stratification.
