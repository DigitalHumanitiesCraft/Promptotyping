# Distillate: AWS Kiro

Kiro is the vendor-side instance of the Spec-Driven-Development wave, an agentic development environment whose spec artefacts and persistent context files are the closest commercial analogue to a Promptotyping document set. Its documentation bounds what the Section 2.5 delimitation may assert about the wave's treatment of data, verification and roles.

## Core statements

- Kiro entered the public record on 2025-07-14 as an agentic development environment positioned beyond unreviewed generation. [[00_representation/documents/sdd-kiro-2026-07-24#^launch]] ^s1
- Its requirements artefact is written in EARS notation, a controlled sentence pattern that binds each requirement to a condition and a system behaviour. [[00_representation/documents/sdd-kiro-2026-07-24#^requirements]] ^s2
- Its design artefact carries data descriptions in the form of data flow diagrams, interfaces, database schemas and API endpoints, all of them descriptions of the system under construction. [[00_representation/documents/sdd-kiro-2026-07-24#^design-data]] ^s3
- The design document is characterised as technical architecture, sequence diagrams and implementation considerations, without a layer describing external sources. [[00_representation/documents/sdd-kiro-2026-07-24#^design-doc]] ^s4
- Approval gates between the phases are the default and their removal is an opt-in convenience for well-understood features. [[00_representation/documents/sdd-kiro-2026-07-24#^approval-gates]] ^s5
- Requirements checking is offered as an analysis for logical inconsistencies, ambiguities, conflicting constraints and gaps, which is internal coherence rather than correspondence to a subject matter. [[00_representation/documents/sdd-kiro-2026-07-24#^analyze-requirements]] ^s6
- Persistent project context is carried by three foundational steering files covering product purpose, technology stack and project structure. [[00_representation/documents/sdd-kiro-2026-07-24#^foundational-files]] ^s7
- None of the three foundational context files describes data sources or their semantics. [[00_representation/documents/sdd-kiro-2026-07-24#^foundational-files]] ^s8
- The documentation names the specification's collaborative reach as product and engineering alignment. [[00_representation/documents/sdd-kiro-2026-07-24#^collaboration]] ^s9
- The pages retrieved state no readership beyond developers and product roles and define no subject-matter verification role. [[00_representation/documents/sdd-kiro-2026-07-24#^negative-audience]] ^s10

## Terms

- **EARS**: Easy Approach to Requirements Syntax, the controlled requirement pattern "WHEN [condition/event] THE SYSTEM SHALL [expected behavior]" that Kiro's requirements artefact uses. [[00_representation/documents/sdd-kiro-2026-07-24#^requirements]]
- **Steering files**: Kiro's persistent workspace knowledge in Markdown, loaded into interactions so conventions need not be restated. [[00_representation/documents/sdd-kiro-2026-07-24#^steering]]

## Open questions

- The design artefact's schemas describe the system's own storage; whether Kiro users in practice use it to describe pre-existing external data is outside what the documentation states.
- The documentation names approval gates without naming who is competent to approve.
